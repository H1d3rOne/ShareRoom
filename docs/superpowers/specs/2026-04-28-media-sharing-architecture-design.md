# ShareRoom 媒体共享架构升级设计

## 背景
当前 ShareRoom 的共享能力主要基于：
- `Socket.IO` 负责房间状态、权限、聊天、控制同步
- 房间内 `WebRTC mesh` 负责实时媒体互传
- `DataChannel + Blob/file` 负责部分文件共享路径
- `iframe` 负责网页共享展示

这套方案在双人或少量成员场景下可用，但在多人房间、弱网、大文件和跨站网页场景下已经暴露出结构性上限：
1. 屏幕共享、游戏共享、网页画面共享会随着房间人数增加而明显变卡。
2. 视频文件共享起播慢，大文件需要长时间接收后才有画面。
3. 网页共享受 iframe 同源策略、CSP `frame-ancestors`、跨域脚本访问限制影响，无法稳定实现“同步画面 + 同步操作”。
4. 现有共享逻辑集中在 `pages/room/room.vue`，共享路径多、状态耦合高，后续继续打补丁的收益会越来越低。

## 目标
1. 将多人实时共享的稳定性提升到可长期演进的产品级架构。
2. 解决屏幕共享、游戏共享、网页画面共享在多人场景下的卡顿和不同步问题。
3. 解决大视频文件共享起播慢、共享者上行压力大、成员重复接收整段文件的问题。
4. 保留当前房间、权限、聊天、游戏等业务模型，尽量避免整站推倒重来。
5. 为后续功能扩展（摄像头、麦克风、更复杂共享控制）保留清晰边界。

## 非目标
1. 本轮不重写整套房间业务和管理员权限系统。
2. 本轮不追求“纯 iframe 兼容所有外部网站并同步网页操作”。
3. 本轮不自研完整 SFU 媒体基础设施。
4. 本轮不一次性重构所有页面样式与非共享相关 UI。

## 方案对比

### 方案 A：继续增强现有 Mesh + DataChannel + iframe
**优点**
- 改动小
- 短期可继续修补局部问题

**缺点**
- 无法解决多人实时共享的带宽放大问题
- 视频文件共享仍受发送端/接收端双边限制
- 网页共享仍会持续撞上浏览器安全边界
- 长期维护成本高

**结论**
仅适合临时止血，不适合作为最终架构。

### 方案 B：全自研 SFU / 全自研媒体平台
**优点**
- 可控性最高
- 长期上限最高

**缺点**
- 工程量和运维成本最高
- 对当前项目体量和节奏不合适
- 首次落地风险高

**结论**
不是当前阶段的最优投入产出比方案。

### 方案 C：LiveKit SFU + 现有业务层 + 服务端视频分发（推荐）
**优点**
- 使用成熟 SFU 解决多人实时共享问题
- 保留现有 Node/Socket.IO 业务层，迁移成本可控
- 视频文件共享可以独立改成更适合的 VOD/Range 路径
- 可将“网页链接共享”和“网页画面共享”拆开处理，避免错误抽象

**缺点**
- 需要引入新的 RTC 基础设施
- 前后端共享链路都需要分层重构

**结论**
这是当前项目最成熟、最稳定、总体效果最好的升级方向。

## 最终架构

### 1. 控制面：保留现有 Node + Socket.IO
继续由现有服务端负责：
- 房间创建、加入、退出
- 超级管理员 / 管理员 / 普通成员权限模型
- 聊天消息
- 游戏邀请、游戏状态同步
- 共享开始/结束状态
- 视频播放控制同步
- 网页链接共享状态

这部分仍然是产品的“业务真相源”。

### 2. 媒体面：引入 LiveKit SFU
以下共享能力统一迁移到 SFU：
- 屏幕共享
- 游戏共享
- 浏览器标签页共享
- 未来扩展的摄像头/麦克风实时流

核心变化：
- 不再由共享者对每个成员建立一份 mesh 上行媒体链路
- 共享者只向 SFU 发一路流
- 其他成员统一从 SFU 订阅

预期收益：
- 多成员时更稳定
- 共享者带宽压力显著下降
- 画面不同步、部分成员卡死的概率显著下降
- 降低当前 `RTCPeerConnection` 重协商复杂度

### 3. 视频文件共享：改为服务端统一分发
视频文件共享不再以 DataChannel/Blob 作为主路径，而改为：
1. 管理员或超级管理员上传视频到服务端临时存储
2. 服务端返回房间可访问的共享 URL
3. 所有成员通过同一 URL 使用原生 `<video>` 播放
4. 播放/暂停/seek/静音等同步仍通过 Socket.IO 广播控制状态

初版采用：
- `HTTP Range` + 原生 `<video>`

后续如有需要，可再演进到：
- HLS
- MSE 分段喂流

### 4. 网页共享：拆分为两种模式

#### 模式 A：网页链接共享
适合：
- 分享网页地址
- 让成员各自打开、浏览
- 保留轻量的 URL / 历史记录 / 前进后退体验

该模式不是“共享统一画面”，而是“共享同一个入口 URL”。

#### 模式 B：网页画面共享
适合：
- 所有人看到相同网页画面
- 主讲人操作网页，其他人同步观看
- 演示网页、SaaS 控制台、在线工具等

实现方式：
- 直接共享浏览器标签页或窗口画面
- 本质归入屏幕共享能力，由 SFU 承载

结论：
- `iframe` 只保留为轻量链接共享能力
- 不再把 `iframe` 当成“通用网页画面同步”的主方案

## 权限模型
现有权限模型保留，不因底层媒体架构变化而改变：
- **超级管理员**：房主，拥有授予/撤销管理员权限，以及全部共享/控制权限
- **管理员**：拥有文件共享、屏幕共享、网页共享、游戏菜单、共享区域全局控制权限
- **普通成员**：仅拥有成员级本地能力（如本地视频播放/暂停、静音/取消静音、全屏等）

迁移要求：
1. 权限判定仍由当前业务层统一管理。
2. LiveKit 仅承担媒体收发，不直接替代房间业务权限真相。
3. 共享控制事件必须继续经过业务层鉴权后再广播。

## 核心数据流

### 屏幕/游戏/网页画面共享
1. 管理员在前端发起共享。
2. 前端向业务服务请求共享会话参数。
3. 业务服务签发 LiveKit 连接参数/令牌。
4. 共享端推送一路媒体流到 LiveKit。
5. 成员从 LiveKit 订阅该共享流。
6. 控制状态（开始、结束、当前共享类型、控制权归属）继续走 Socket.IO。

### 视频文件共享
1. 管理员选择本地视频文件。
2. 前端上传到业务服务临时存储。
3. 服务端生成共享媒体记录和房间内访问 URL。
4. 成员收到共享开始事件后，统一从服务端拉取视频。
5. 管理员控制播放进度时，通过 Socket.IO 广播同步状态。
6. 普通成员的本地播放/静音仅影响本地播放器状态，不回写全局。

### 网页链接共享
1. 管理员输入 URL。
2. 业务层记录共享 URL、历史栈、当前索引。
3. 各成员本地加载该 URL（允许失败，失败不影响房间状态）。
4. 如需统一观看画面，改用“网页画面共享”模式，而不是继续要求 iframe 同步站内操作。

## 组件拆分建议

### 前端
将当前 `pages/room/room.vue` 中的共享逻辑逐步拆分为：
- `share/permissions`：共享控制权限判定
- `share/livekit-session`：实时共享会话接入与状态管理
- `share/video-vod`：视频文件上传、URL 播放、同步控制
- `share/webpage-link`：网页链接共享、历史记录、地址切换
- `share/shared-stage`：共享区域渲染与全屏能力
- `share/share-toolbar`：共享操作按钮区

### 后端
将当前 `server/server.js` 中的共享能力逐步拆分为：
- `room-state`：房间状态与参与者管理
- `room-permissions`：超级管理员/管理员鉴权
- `share-session`：共享开始/结束/同步状态
- `livekit-token`：实时共享鉴权和 token 下发
- `video-upload`：视频上传
- `video-streaming`：视频 Range 响应
- `webpage-share`：网页链接共享状态

本轮不要求一次性拆完，但后续每个实施阶段都应向这个边界收敛。

## 错误处理与回退策略
1. **LiveKit 不可用**：共享入口应明确提示，不静默失败；必要时允许回退到“仅链接共享”或“不可用提示”。
2. **视频上传失败**：不创建共享记录，前端保留重试入口。
3. **视频拉流失败**：成员端显示可恢复错误，不影响房间其他业务。
4. **某网站拒绝 iframe 嵌入**：网页链接共享保留失败态提示，建议用户改用网页画面共享。
5. **共享者离开房间**：保持现有语义，关闭对应共享并广播房间提示。
6. **管理员权限变化**：业务层立即广播，前端实时刷新控制能力，不依赖页面刷新。

## 测试策略
1. **后端单元测试**
- 权限鉴权
- 共享会话状态流转
- 视频上传元数据记录
- 视频 Range 响应

2. **前端单元测试**
- 超级管理员 / 管理员 / 普通成员共享能力矩阵
- 视频共享同步与本地控制分离逻辑
- 网页链接共享历史栈与环形切换逻辑
- 共享区域全屏/关闭按钮行为

3. **联调测试**
- 2/4/8 人房间屏幕共享
- 管理员共享后超级管理员接管控制
- 大视频上传后多成员快速起播
- 网页链接共享失败时 UI 提示正确

4. **回归重点**
- 现有聊天、成员列表、管理员授予/撤销不回退
- 游戏菜单与游戏阶段全屏/关闭行为不回退
- 房主退出/最后管理员退出的房间语义不回退

## 分阶段实施建议

### 阶段 1：实时共享迁移到 LiveKit
范围：
- 屏幕共享
- 游戏共享
- 浏览器标签页共享

目标：
先解决多人房间实时共享卡顿、画面不同步、连接脆弱的问题。

### 阶段 2：视频文件共享切换到服务端统一分发
范围：
- 视频上传
- 临时存储
- Range 播放
- 全局播放控制同步

目标：
解决大文件起播慢、共享者上行压力大、成员重复接收的问题。

### 阶段 3：网页共享重构为双模式
范围：
- 链接共享
- 网页画面共享入口切换
- iframe 失败态提示优化

目标：
停止把 iframe 当成通用同步网页方案，明确产品语义。

## 下一步约束
这个设计覆盖多个相对独立的子系统，因此**不应直接写成一个超大 implementation plan 并一次性开做**。后续应拆成至少三个独立实施计划：
1. LiveKit 实时共享接入计划
2. 视频文件服务端分发计划
3. 网页共享双模式重构计划

推荐的实施顺序是：
1. 先做 LiveKit 实时共享接入
2. 再做视频文件分发
3. 最后做网页共享双模式收口
