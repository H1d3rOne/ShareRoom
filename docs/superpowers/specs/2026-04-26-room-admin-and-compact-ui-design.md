# 房间管理员分级与紧凑 UI 调整设计

日期：2026-04-26
涉及文件：`pages/room/room.vue`、`server/server.js`

## 目标

在现有房间页基础上完成两类调整：

1. 继续收紧聊天输入区与成员列表的视觉占用。
2. 把当前“单管理员”模型扩展为“超级管理员 + 普通管理员”模型，并修正管理员授予与离房逻辑。

本次需求包含以下明确目标：

- 聊天输入框左侧 3 个按钮再紧凑一点。
- “发送”按钮再紧凑一点，让输入框更宽。
- 房间成员区域的用户显示整体更小一点。
- “授予管理员”应当真正生效。
- 创建房间的人是超级管理员，其他被授予的人是普通管理员。
- 只有超级管理员拥有“授予管理员权限”的能力。
- 超级管理员退出时，如果房间里还有普通管理员，房间继续保留。
- 当最后一个管理员体系成员离开时，房间关闭并向房内成员提示。
- 文件共享中的视频共享界面去掉重复的“关闭”按钮，只保留一个“关闭共享”按钮。

## 当前实现与问题

### 1. 权限模型问题

当前后端使用的是单管理员模型：

- `room.adminSocketId`
- `room.adminClientId`

前端按钮文案虽然是“授予管理员”，但后端事件名与行为实际是 `transfer-admin`，本质是**把唯一管理员身份转移给别人**，不是给房间新增一个管理员。

因此，它与当前需求冲突：

- 不能同时存在超级管理员与普通管理员
- 也无法表达“只有创建者才可以授予管理员”

### 2. 离房逻辑问题

当前 `leave-room` 会以 `releaseAdmin: true` 的方式处理管理员离房。只要当前管理员离开，就会触发房间关闭广播。

这与新规则不一致：

- 超级管理员退出时，如果房间里还有普通管理员，房间应继续存在
- 只有当最后一个管理员体系成员离开时，房间才关闭

### 3. UI 问题

当前聊天输入区和成员区虽然已经做过一轮紧凑化，但仍然存在：

- 左侧设备按钮仍偏松
- “发送”按钮仍占用相对更多空间
- 成员卡片的头像、字号、标签和按钮仍略大
- 视频共享区域同时出现“关闭”和“关闭共享”两个关闭入口，重复

## 方案

采用“在现有结构上做增量扩展”的方案，不重做房间体系。

### 方案选择

本次采用：**在现有单管理员结构上扩展成“超级管理员 + 普通管理员”双层权限**。

理由：

- 与现有 `room.vue` / `server.js` 结构最兼容
- 改动集中，风险最低
- 能满足当前产品规则，不需要为本次需求引入更重的全量角色系统重构

## 权限设计

### 1. 角色定义

房间中存在三类身份：

#### 超级管理员

- 由房间创建者担任
- 在房间生命周期内保留该身份
- 拥有普通管理员全部能力
- 比普通管理员额外多一个能力：**授予其他成员管理员权限**

#### 普通管理员

- 由超级管理员授予
- 保留现有管理员能力，例如：
  - 文件/屏幕/网页共享
  - 关闭共享
  - 与当前管理员能力等价的日常管理能力
- **不拥有授予管理员权限的能力**

#### 普通成员

- 保持现有普通成员能力

### 2. 数据模型调整

后端房间状态由“单管理员字段”扩展为：

- `superAdminSocketId`
- `superAdminClientId`
- `adminSocketIds`（普通管理员集合）

前端参与者序列化信息应支持区分：

- `isSuperAdmin`
- `isAdmin`

其中：

- 超级管理员在前端也视为管理员体系成员
- 普通管理员只拥有 `isAdmin`
- 超级管理员同时拥有 `isSuperAdmin`

### 3. 授权行为

当前 `transfer-admin` 的语义将调整为真正的“授予管理员”：

- 只有超级管理员可以触发
- 被授予成员成为普通管理员
- 超级管理员自身身份不变化
- 不发生“转让超级管理员”行为

本次不做：

- 撤销普通管理员
- 转移超级管理员
- 多级授权链

## 房间生命周期设计

### 1. 超级管理员退出

超级管理员退出房间时：

- 如果房间里还有普通管理员，房间继续保留
- 不自动转让超级管理员身份
- 房内广播系统提示，说明超级管理员已离开，但房间继续由管理员维持

### 2. 普通管理员退出

普通管理员退出时：

- 从普通管理员集合中移除
- 房间继续保留（只要仍有管理员体系成员存在）
- 房内广播系统提示

### 3. 房间关闭条件

仅在以下场景关闭房间：

- 超级管理员已不在房间中
- 普通管理员集合也为空
- 即：**最后一个管理员体系成员离开时**

关闭时：

- 广播 `room-closed`
- 给出明确关闭提示
- 前端收到后提示并返回首页

### 4. 创建房间后的初始状态

创建房间的人首次加入时：

- 自动成为超级管理员
- 房间内没有普通管理员

后续同一创建者重连时，应恢复其超级管理员身份，而不是降级为普通成员。

## 前端 UI 设计

### 1. 聊天输入区进一步紧凑化

继续在当前样式基础上做小幅压缩：

- 左侧 3 个设备按钮进一步缩小间距
- 按钮尺寸与内边距继续小一档，但保留可点击面积
- “发送”按钮左右 padding 缩小，宽度更紧凑
- 输入框因此获得更多横向空间

### 2. 成员列表进一步紧凑化

只缩小成员卡片内部元素，不改面板结构：

- 头像略缩小
- 名字字号略缩小
- 标签字号略缩小
- 操作按钮尺寸与间距略缩小
- 成员卡片 padding 略缩小

目标是在同一视野里显示更多成员信息，但不牺牲可读性。

### 3. 成员身份展示

成员列表身份标签调整为：

- 超级管理员：显示“超级管理员”
- 普通管理员：显示“管理员”
- 自己但非管理员：显示“我”
- 普通成员：继续显示现有连接状态

“授予管理员”按钮的显示规则：

- 仅超级管理员可见
- 不显示给自己
- 不显示给已经是普通管理员或超级管理员的人

### 4. 视频共享关闭按钮

视频共享区删除重复关闭入口：

- 去掉视频控制区中的“关闭”按钮
- 保留底部统一的“关闭共享”按钮

这样可避免两个关闭入口并存造成的重复与混淆。

## 前端交互与提示文案

需要同步调整系统消息与界面提示，使其符合新权限模型，例如：

- 某成员被授予管理员权限
- 超级管理员已离开，房间继续由管理员维持
- 最后一个管理员离开，房间已关闭

原有“管理员转移”语义相关提示，应改为“授予管理员”语义。

## 涉及文件

### `server/server.js`

负责：

- 房间角色状态维护
- 参与者序列化
- 管理员授予逻辑
- 管理员离房后的房间存活/关闭逻辑
- 房间关闭广播文案

### `pages/room/room.vue`

负责：

- 聊天输入区进一步紧凑化
- 成员区进一步紧凑化
- 超级管理员 / 管理员标签显示
- “授予管理员”按钮显示条件
- 前端管理员相关提示文案更新
- 去掉视频共享中的重复“关闭”按钮

## 不做的内容

本次明确不包含：

- 撤销普通管理员
- 超级管理员转移
- 新增更复杂的角色权限配置系统
- 重做成员面板整体布局
- 重构聊天消息数据结构
- 重构文件共享协议，仅移除重复按钮入口

## 验收标准

完成后应满足：

1. 超级管理员可以给其他成员授予管理员权限。
2. 普通管理员不显示“授予管理员”按钮。
3. 超级管理员拥有普通管理员全部能力，并额外拥有授予管理员能力。
4. 超级管理员退出时，如果房间里仍有普通管理员，房间继续存在。
5. 当最后一个管理员体系成员离开时，房间关闭并提示。
6. 聊天输入区左侧按钮比现在更紧凑，“发送”按钮也更紧凑，输入框更宽。
7. 成员列表中的头像、名字、标签、按钮整体更小更紧凑。
8. 视频共享时只保留一个“关闭共享”按钮。
9. 相关系统提示文案与权限模型保持一致，不再出现“管理员转移”语义误导。
