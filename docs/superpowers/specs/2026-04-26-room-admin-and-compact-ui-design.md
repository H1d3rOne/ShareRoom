# 房间管理员分级、共享控制分层与紧凑 UI 调整设计

日期：2026-04-26  
涉及文件：`pages/room/room.vue`、`server/server.js`、`tests/server/room-admin-roles.test.mjs`、`tests/ui/room-compact-ui.test.mjs`

## 目标

在现有房间页基础上完成三类调整：

1. 继续收紧聊天输入区与成员列表的视觉占用。
2. 把房间权限模型稳定为“超级管理员 + 管理员 + 普通成员”。
3. 把“共享控制”明确拆分为“管理员全局控制”与“普通成员本地控制”。

本次需求包含以下明确目标：

- 房主是**超级管理员**。
- 超级管理员授予给其他用户的是**管理员权限**，不是转移房主身份。
- 授予管理员后，房主依然是超级管理员，并保留授予管理员的能力。
- 管理员获得后拥有：
  - 文件共享
  - 屏幕共享
  - 网页共享
  - 游戏菜单
  - 共享区域的完整控制权限
- 管理员与超级管理员的差别只有一点：
  - **只有超级管理员可以授予管理员权限**
- 超级管理员与管理员在成员列表中要明确区分：
  - 房主显示“超级管理员”
  - 被授予者显示“管理员”
- 视频共享全屏时，进度条应随可用宽度一起扩展。
- 管理员对共享区域享有**全局控制权限**。
- 普通成员不能改全局共享状态，但都可以控制**自己本地**视频的：
  - 播放 / 暂停
  - 声音开启 / 关闭
  - 全屏
- 普通成员本地暂停视频后，再次播放时，应自动追上当前管理端同步进度，再恢复自己的本地播放。

## 当前实现与问题

### 1. 角色模型已基本成型，但权限边界还不完整

当前房间角色模型已经支持：

- `superAdminSocketId`
- `superAdminClientId`
- `adminSocketIds`
- 前端参与者字段：
  - `isSuperAdmin`
  - `isAdmin`

管理员授予也已经从“转移管理员”改成了“授予管理员”。

但当前前端能力边界还不完全符合新规则：

- `canShare` 已经绑定到 `isAdmin`，这符合“管理员可共享”
- 但“游戏菜单”按钮还没有严格限制为管理员
- “共享控制”仍主要围绕“共享方 / 远控方”判断，未完整表达“管理员全局、普通成员仅本地”的新规则

### 2. 视频控制仍偏向单层控制模型

当前视频共享控制栏主要使用 `canControlShare` 判断：

- 能控制的人可以播放/暂停/拖动进度/静音
- 不能控制的人整体被禁用

这与新规则不一致：

- 管理员体系成员应拥有**全局控制**
- 普通成员不应被完全禁用，而应拥有**本地播放/暂停与本地静音**能力
- 普通成员恢复播放时，还要自动追到当前全局同步时间

### 3. 全屏态缺少专门的视频控制布局

当前“全屏”只是让 `shared-stage` 进入全屏：

- 进度条本身仍沿用普通态布局
- 没有明确的全屏态扩展规则

这与“共享视频全屏时进度条随宽度扩展”的体验目标不一致。

## 方案

继续采用“在现有结构上增量扩展”的方案，不重做房间体系，不拆大型文件。

### 方案选择

本次采用：**角色模型保持现状，在前端补上能力分层与视频控制分层。**

理由：

- 后端角色数据结构已足够表达“超级管理员 + 管理员”
- 主要缺口已经从“谁是管理员”转为“管理员和普通成员各能做什么”
- 在现有 `room.vue` 上补充能力计算与视频控制分支，风险低、改动集中

## 权限设计

### 1. 角色定义

#### 超级管理员

- 由房间创建者担任
- 在房间生命周期内保留该身份
- 拥有管理员全部能力
- 比管理员额外多一个能力：**授予其他成员管理员权限**

#### 管理员

- 由超级管理员授予
- 拥有以下能力：
  - 文件共享
  - 屏幕共享
  - 网页共享
  - 打开游戏菜单
  - 关闭共享
  - 共享区域全局控制
  - 屏幕共享远控相关能力
- 不拥有授予管理员权限的能力

#### 普通成员

- 不具备共享发起权
- 不具备共享区域全局控制权
- 在视频共享场景下，保留对**个人端**的本地控制：
  - 本地播放 / 暂停
  - 本地静音 / 取消静音

### 2. 能力模型

前端能力划分为以下几类：

- `canGrantAdmin`
  - 仅超级管理员为 `true`
- `canShare`
  - 超级管理员 / 管理员为 `true`
- `canOpenGameMenu`
  - 超级管理员 / 管理员为 `true`
- `canGlobalControlShare`
  - 超级管理员 / 管理员为 `true`
- `canLocalControlSharedVideo`
  - 所有人在视频共享场景都可为 `true`

其中：

- `canGlobalControlShare` 用于“改变房间共同看到的共享状态”
- `canLocalControlSharedVideo` 只用于本地 `<video>` 控制，不发全局同步事件

### 3. 授权行为

管理员授予语义保持为：

- 只有超级管理员可以触发
- 被授予成员成为管理员
- 超级管理员自身身份不变化
- 不发生“房主转让”
- 不发生“管理员链式授权”

## 共享控制设计

### 1. 管理员的全局控制

管理员体系成员（超级管理员 + 管理员）对共享区拥有完整全局控制权：

#### 视频共享

- 全局播放 / 暂停
- 全局拖动进度
- 全局同步当前播放时间
- 全局维持心跳同步

#### 图片共享

- 全局放大 / 还原

#### 屏幕共享

- 远控申请与授权链路
- 远控指针
- 远控命令

#### 网页共享

- 保持管理员体系成员可发起 / 关闭共享
- 共享呈现仍由共享方广播给房间

### 2. 普通成员的本地视频控制

普通成员在视频共享场景下只允许控制自己的本地播放器：

- 点击“播放 / 暂停”只影响本地视频元素
- 点击“声音开 / 关”只影响本地 `muted`
- 点击“全屏”只影响自己的本地观看界面
- 不发送 `share-control`
- 不改变房间内其他用户看到的播放状态

### 3. 普通成员恢复播放时的追帧规则

当普通成员本地暂停后再次播放时：

1. 先读取当前房间全局 `sync`
2. 计算管理端当前应处的播放时间
3. 先把本地视频跳到该时间
4. 再只恢复本地播放

这样保证：

- 普通成员可以自由暂停自己的观看
- 恢复播放时不会停留在旧进度
- 房间全局同步仍以管理员控制为准

### 4. 控制栏交互分层

视频控制栏应在同一块 UI 中表达两层能力：

- 管理员看到的是**全局控制栏**
  - 播放 / 暂停
  - 进度拖动
  - 静音按钮（本地听感）
  - 全屏
- 普通成员看到的是**本地控制栏**
  - 本地播放 / 暂停
  - 本地静音
  - 本地全屏
  - 不允许拖动全局进度条

为避免误解，普通成员的进度条可以：

- 继续显示当前全局进度
- 但不可拖动

## 全屏态设计

### 1. 共享视频全屏

当共享舞台进入全屏后：

- 视频控制区应拉伸到更宽布局
- 进度条成为主要横向元素
- 时间标签与按钮保持紧凑，不抢占宽度

### 2. 样式目标

全屏态下：

- `.video-control-panel` 或其内部布局切换为更宽的 grid/flex 分配
- `.progress-slider` 占据更多剩余宽度
- 控制按钮维持当前紧凑尺寸，不额外放大

## 前端展示与文案

### 1. 成员身份展示

成员列表身份标签继续使用：

- 超级管理员：显示“超级管理员”
- 管理员：显示“管理员”
- 自己但非管理员：显示“我”
- 普通成员：显示连接状态

### 2. 共享权限提示

需要把界面文案从“只有房主可共享”修正为“管理员体系成员可共享”，例如：

- 空状态提示
- 远控提示
- 共享控制说明

避免把“超级管理员”和“管理员”混成一个概念。

### 3. 游戏菜单权限

游戏菜单入口应调整为：

- 超级管理员 / 管理员可见且可点
- 普通成员不显示或禁用（本次优先采用不显示）

- 只有超级管理员可以授予管理员

本次不新增复杂后端角色层级，只补前端权限消费与视频控制分层时需要的契约确认。

### 2. 前端结构与行为测试

前端测试需要补充：

- 游戏菜单按钮受管理员权限控制
- 管理员共享提示文案不是“只有房主可共享”
- 视频控制栏存在管理员全局控制与普通成员本地控制的判定逻辑
- 普通成员恢复播放时会追到当前全局同步时间
- 全屏态存在进度条扩展样式
